What is DynamoDB?

Amazon DynamoDB is a fully managed, serverless, NoSQL key-value and document database that delivers single-digit millisecond performance at any scale.

Foundational properties:

  • Fully managed
  • Serverless
  • NoSQL
  • Key-value and document
  • Performance in single-digit milliseconds
  • Durability across 3 AZs within a Region
  • Availability 99.99% SLA; 99.999% for Global Tables
  • Scales to any size
  • Regional service
  • No resource-based policies

Primary keys

The primary key uniquely identifies each item in a table. It is chosen at table creation and cannot be changed afterward.

Two primary key types

TypeCompositionHow DynamoDB uses it
Partition key onlySingle attributeHash determines partition
Partition key + Sort keyTwo attributesHash determines partition; sort key orders within it

Partition key design

  • High cardinality
  • Even distribution
  • Avoid hot keys like limited-status values or traffic-skewed dates

Hot partition = throttling.

Sort key

Enables range queries, prefix queries, and one-to-many modeling within a partition.

Read Capacity Units (RCU):

Read typeItem sizeRCU cost

TransactionalUp to 4 KB2 RCU

Write Capacity Units (WCU):

Write typeItem sizeWCU cost
StandardUp to 1 KB1 WCU
TransactionalUp to 1 KB2 WCU

Items larger than the unit size are rounded up.

Switching modes

  • Provisioned → On-demand: anytime
  • On-demand → Provisioned: anytime
  • Provisioned → On-demand is limited to 4 times in 24 hours

DynamoDB Standard-IA table class

Lower storage cost, higher per-request cost. Best when storage dominates and access is infrequent.

GSI throttling

If a GSI runs out of write capacity, the base table is also throttled.

Read consistency models

ModelBehaviorCostUse case
Eventually consistentMay return slightly stale data0.5 RCU per 4 KBDefault, most workloads
Strongly consistentReturns the latest data1 RCU per 4 KBWhen latest data is required
  • GSIs support only eventual consistency.
  • LSIs support both eventual and strong consistency.
  • Global Tables are eventually consistent across Regions by default.

DynamoDB Streams

DynamoDB Streams captures a time-ordered sequence of item-level modifications over the last 24 hours.

Time to Live (TTL)

TTL automatically deletes expired items from a table without consuming write capacity.

  1. Choose a TTL attribute storing a Unix epoch timestamp (Number type).
  2. Enable TTL on the table.
  3. DynamoDB deletes expired items within a few days.
  • TTL deletes do not consume WCU in the source Region.
  • Deleted items appear in DynamoDB Streams.
  • In Global Tables, TTL deletes replicate and consume replicated WCU.

Transactions

DynamoDB supports ACID transactions across multiple items and tables within a single Region.

OperationDescription
TransactWriteItemsAtomic write of up to 100 items
TransactGetItemsAtomic read of up to 100 items
  • Maximum 100 items per transaction
  • Maximum 4 MB per transaction
  • Cannot target the same item twice in one transaction
  • Transactions consume 2x RCU/WCU
  • Not supported across Regions in Global Tables

Backups

On-demand backup

  • Full backup of a table at any point in time
  • No performance impact
  • Retained until you delete it
  • Restores to a new table

Point-in-Time Recovery (PITR)

  • Continuous backups with 35-day recovery window
  • Restore to any second within the last 35 days
  • Must be explicitly enabled
  • Restores to a new table

Security

Encryption at rest

  • All DynamoDB tables are encrypted at rest and this cannot be disabled.
  • Key options: AWS owned, AWS managed, or customer managed KMS key.
  • You can change key type up to 4 times per 24 hours.

Encryption in transit

All DynamoDB API calls use HTTPS/TLS by default.

Access control

  • IAM policies control table access.
  • dynamodb:LeadingKeys restricts access by partition key values.
  • dynamodb:Attributes restricts which attributes can be accessed.
  • VPC endpoints provide private access from VPCs.

DynamoDB and VPC access

DynamoDB is a public service. For private access from a VPC use:

  • Gateway VPC endpoint (recommended, free)
  • Interface VPC endpoint (costs money)

Lambda in a VPC should typically use a gateway VPC endpoint to avoid a NAT Gateway.

Conditional writes and optimistic locking

Conditional writes

Add conditions so a write only succeeds if an attribute has an expected value.

Optimistic locking

Use a version number attribute and conditionally update only if the expected version matches.

Scan vs. Query

OperationBehaviorEfficiency
QueryFinds items by partition key (and optionally sort key conditions)Efficient
ScanReads every item in the table or indexInefficient
  • Prefer Query over Scan.
  • Parallel Scan can speed up a scan but consumes more RCU.
  • Filter expressions do not reduce RCU consumption.

Patterns and anti-patterns

When to use DynamoDB

  • Known access patterns with key-based lookups
  • High-scale, low-latency applications
  • Session stores, carts, user profiles
  • Time-series data
  • Event-driven architectures

When NOT to use DynamoDB

  • Complex joins and ad-hoc queries
  • Full-text search
  • Large cross-table transactions
  • OLAP analytics workloads

Common exam traps and edge cases

  1. Maximum item size is 400 KB.
  2. LSI must be created at table creation.
  3. LSI has a 10 GB item collection limit.
  4. GSI only supports eventually consistent reads.
  5. GSI throttling throttles the base table.
  6. Filter expressions do NOT reduce RCU.
  7. Transactions cost 2x.
  8. DynamoDB Streams retention is only 24 hours.
  9. Global Tables transactions are Region-local only.
  10. PITR restores to a NEW table.
  11. DAX only supports eventually consistent reads.
  12. On-demand mode adapts to 2x previous peak.
  13. No resource-based policies.

What is EFS?

Amazon Elastic File System (EFS) is a fully managed, serverless, elastic NFS file system that can be shared across multiple compute resources simultaneously. It uses the NFS v4.1 protocol and provides standard file system semantics.

The one-line decision rule: Need shared file access across multiple instances? → EFS. Need a high-performance disk for one instance? → EBS. Need to store objects accessible via API? → S3.

Core concepts

File systems

An EFS file system is the top-level resource.

TypeDurabilityAZ scopeCostUse case
Regional11 ninesMultiple AZsHigherProduction HA workloads
One Zone11 nines within that AZSingle AZ~47% cheaperDev/test, re-creatable data

If the single AZ of a One Zone file system is destroyed, data is lost. Use Regional for anything critical.

Mount targets

A mount target is a network endpoint in a specific AZ that compute resources use to connect to the file system.

  • Regional: Create one mount target per AZ you use.
  • One Zone: Only one mount target in the same AZ.
  • Each mount target uses a subnet and security group.
  • Security groups must allow inbound TCP 2049.

How mounting works

# Install the EFS mount helper
sudo yum install -y amazon-efs-utils

# Mount using the helper
sudo mount -t efs -o tls fs-0123456789abcdef0:/ /mnt/efs

# Or mount using standard NFS
sudo mount -t nfs4 -o nfsvers=4.1 fs-0123456789abcdef0.efs.us-east-1.amazonaws.com:/ /mnt/efs

Who can mount EFS: EC2, ECS/Fargate, EKS, Lambda, and on-premises servers over Direct Connect or VPN.

Storage classes and lifecycle management

Storage classes

Storage ClassLatencyCostUse case
EFS StandardSub-millisecondHighestFrequently accessed data
EFS Infrequent Access (IA)Low double-digit ms~92% cheaperAccessed a few times per quarter
EFS ArchiveLow double-digit ms~50% cheaper than IAAccessed a few times per year

Available for both Regional and One Zone file systems.

Lifecycle policies (EFS Intelligent-Tiering)

  1. Transition to IA
  2. Transition to Archive
  3. Transition back to Standard (optional)
  • Operates at the file level.
  • Files smaller than 128 KB are not transitioned.
  • Accessing IA and Archive incurs a per-GB retrieval fee.
  • Archive requires Elastic throughput.

Performance modes

Elastic throughput

  • Recommended for most workloads.
  • Automatically scales up to 1,500 MiB/s.
  • No burst credits to manage.
  • Reads are metered at one-third the rate of writes.

Bursting throughput

  • Base rate: 50 KiB/s per GiB of Standard storage.
  • Small file systems get very little baseline throughput.

Provisioned throughput

  • You specify exact throughput.
  • Must wait 24 hours after increasing before decreasing.

EFS Access Points

An access point is an application-specific entry point into an EFS file system.

  • Root directory override
  • POSIX user/group override
  • Automatic directory creation

Useful for isolating multiple applications on one file system and for Lambda.

Security

Encryption

At restKMS keyMust be enabled at creation; cannot be enabled later

Network security

Security groups on mount targets must allow inbound TCP 2049.

IAM and file system policies

  • IAM policies control API actions.
  • File system policies control NFS-level access.
  • POSIX permissions apply to files and directories.

All layers must allow access.

On-premises access

EFS can be mounted from on-premises servers over Direct Connect or VPN.

  1. Establish Direct Connect or VPN to the VPC.
  2. Create a mount target in the VPC.
  3. Mount the file system using the mount target IP.

For migrations, use AWS DataSync.

EFS replication

EFS supports automatic replication to another Region or another AZ in the same Region.

  • Destination is a read-only replica.
  • Asynchronous with an RPO of minutes.
  • Only one replication configuration per file system.
  • Promote destination to read-write during failover.

EFS with Lambda

Lambda functions can mount EFS when configured in a VPC.

Requirements:

  1. Lambda must be attached to a VPC.
  2. An EFS access point must be configured.
  3. Lambda security group must allow outbound to the mount target on port 2049.
  4. Mount target security group must allow inbound from Lambda on port 2049.

Lambda in a VPC loses internet access, so it may also need NAT or VPC endpoints.

Cost

EFS pricing model

Pay-per-use with no provisioning required. Pricing varies by storage class and file system type.

Relative cost

Storage classRelative cost
EFS Standard (Regional)Highest
EFS Standard (One Zone)~47% cheaper than Regional Standard
EFS IA (Regional)~92% cheaper than Regional Standard
EFS IA (One Zone)~92% cheaper than One Zone Standard
EFS Archive~50% cheaper than IA

Cost optimization strategies

  1. Enable lifecycle policies.
  2. Use One Zone for non-critical data.
  3. Use Elastic throughput instead of Provisioned when possible.
  4. Monitor with CloudWatch.
  5. Use access points instead of unnecessary separate file systems.

Common exam traps and edge cases

  1. EFS is Linux only.
  2. Encryption at rest cannot be enabled later.
  3. Bursting throughput is terrible for small file systems.
  4. Performance mode cannot be changed (mostly).
  5. EFS is not S3.
  6. Security group must allow port 2049.
  7. Lambda + EFS requires VPC + access point.
  8. One Zone ≠ Multi-AZ.
  9. Files < 128 KB are not transitioned.
  10. Archive class locks you to Elastic throughput.
  11. FSx for Lustre is not EFS.
  12. EFS replication destination is read-only.

Final mental model

Every EFS/FSx exam question can be broken down with this sequence:

  1. Does it need to be shared across multiple instances?
  2. What OS?
  3. What protocol?
  4. What performance is needed?
  5. What availability is needed?
  6. What is the access pattern?
  7. What is the cost priority?

If you can answer these seven questions for any scenario, you can answer any EFS/FSx question on the SAA-C03.

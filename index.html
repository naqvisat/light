AWS IAM: The Complete Guide for SAA-C03

Everything you need to know about Identity and Access Management — definitions, examples, edge cases, evaluation logic, hard limits, and exam traps — in one place.

What is IAM, really?

IAM stands for Identity and Access Management. It is the control plane that governs every interaction inside AWS. Every time anything tries to act in AWS - a human clicking in the console, an EC2 instance reading from S3, a Lambda function writing to DynamoDB, one AWS account reaching into another - IAM answers two questions:

  1. Who is making this request? (Authentication)
  2. Is this request allowed? (Authorization)

The foundational rule: AWS services do not trust each other automatically. Access must be granted explicitly. Nothing can do anything unless a policy somewhere says it can.

IAM is a global service. It is not scoped to a Region. Users, roles, and policies you create are available across all Regions in your account.

The core IAM objects

The root user

Every AWS account has exactly one root user. This is the email address used to create the account. The root user has complete, unrestricted access to everything in the account - it cannot be limited by IAM policies, permission boundaries, or SCPs (though SCPs can restrict the root user in member accounts of an Organization for most actions).

Best practices for the root user:

  • Enable MFA immediately. Use a hardware MFA device if possible.
  • Do not create access keys for the root user.
  • Do not use the root user for day-to-day tasks.
  • Use it only for the few tasks that require root: changing account settings, closing the account, restoring IAM user permissions, changing the support plan, enabling certain S3 features like MFA Delete.

Exam trap: Some tasks can only be done by the root user. The exam may present a scenario where an administrator with full IAM permissions still cannot perform an action. The answer is often "use the root user."

Users

A user is a long-term identity, usually representing a human. Users can have:

  • A console password (for web console access)
  • Access keys (for CLI/API access) - a user can have a maximum of 2 access keys
  • Both, or neither

Users authenticate with their credentials and then IAM evaluates their policies to decide what they can do.

When to use users vs. roles: Users are appropriate for human identities who need long-term access. For applications, services, and cross-account access, always prefer roles. In modern AWS, even human access increasingly uses federation through IAM Identity Center rather than IAM users.

Groups

A group is a collection of users. You attach policies to a group, and every user in that group inherits those permissions.

Key rules:

  • Groups are not identities. You cannot sign in as a group.
  • Groups cannot be nested. A group cannot contain another group.
  • A user can belong to a maximum of 10 groups.
  • There is no "all users" group by default. If you want one, you must create it.
  • Groups cannot be referenced as a Principal in a policy. You cannot grant cross-account access to a group.

Roles

A role is a temporary identity that can be assumed. This is the most important IAM object in modern AWS architecture because roles are how services, applications, and cross-account access work.

How roles differ from users:

  • Roles do not have long-term passwords or access keys.
  • Instead, AWS Security Token Service (STS) issues temporary credentials when something assumes a role.
  • Temporary credentials consist of an access key ID, a secret access key, and a session token, and they expire automatically.
  • Multiple entities can assume the same role simultaneously.

Who can assume a role:

  • AWS services (EC2, Lambda, ECS, etc.)
  • IAM users in the same or different accounts
  • IAM roles in the same or different accounts
  • Federated users (SAML, OIDC, web identity)
  • The root user

Every role has exactly two types of policy:

  1. Trust policy - defines who is allowed to assume the role (the lock on the cabinet)
  2. Permissions policy - defines what the role can do once assumed (what is inside the cabinet)

Critical rule: The trust policy always lives on the role, never on the caller. EC2 does not say "I trust this role." The role says "I trust EC2."

Instance profiles

An instance profile is a container that connects an IAM role to an EC2 instance. When you create a role for EC2 in the console, AWS automatically creates a matching instance profile with the same name. Through the CLI or API, you must create the instance profile separately and add the role to it.

  • An instance profile can contain exactly one role.
  • An EC2 instance can have exactly one instance profile attached.

Policies

A policy is a JSON document that defines permissions. Policies answer: "Can this identity perform this action on this resource under these conditions?"

Policy types - the complete taxonomy

Understanding the different policy types and how they interact is critical for the exam.

1. Identity-based policies

Attached to users, groups, or roles. These are the "normal" IAM policies that grant day-to-day permissions.

Two sub-types:

  • Managed policies - standalone policy objects that can be attached to multiple identities.
    • AWS managed policies - created and maintained by AWS (e.g., AmazonS3ReadOnlyAccess). AWS updates them when new features launch.
    • Customer managed policies - created and maintained by you. Preferred over inline policies because they are reusable, versionable (up to 5 versions), and centrally manageable.
  • Inline policies - embedded directly inside a single user, group, or role. They maintain a strict one-to-one relationship with the identity. If you delete the identity, the inline policy is also deleted. Use inline policies when you intentionally want to ensure a policy is never accidentally attached to the wrong identity.

2. Resource-based policies

Attached to the resource itself rather than to the caller. The resource declares who may access it.

Key characteristics:

  • They include a Principal element (identity-based policies do not, because the principal is the identity the policy is attached to).
  • They are always inline - you cannot create a standalone managed resource-based policy.
  • Not every AWS service supports resource-based policies.

Services that support resource-based policies (exam-relevant):

3. Permission boundaries

A permission boundary is a managed policy attached to a user or role that sets the maximum permissions that identity can ever have. It does not grant permissions - it is a ceiling.

How it works: The effective permissions of the identity are the intersection of the permission boundary and the identity-based policy. If the identity policy allows s3:* but the permission boundary only allows s3:GetObject, the effective permission is s3:GetObject.

Use case: Delegated administration. You want to allow developers to create their own IAM roles, but you do not want them to create roles more powerful than their own. You attach a permission boundary that caps what any role they create can do.

Exam trigger: "Allow a team to create IAM roles without exceeding a defined set of permissions."

4. Service Control Policies (SCPs)

SCPs are part of AWS Organizations. They set the maximum permissions for accounts or organizational units (OUs).

Critical rules:

  • SCPs do not grant permissions. They only restrict.
  • SCPs do not affect the management account (the account that created the Organization). They only affect member accounts.
  • SCPs affect all users and roles in the member account, including the root user of that member account (for most actions).
  • SCPs do not affect service-linked roles.
  • An explicit deny in an SCP overrides any allow in any IAM policy in that account.

Exam trigger: "Prevent any user in the Production OU from terminating EC2 instances" - answer is SCP with a deny on ec2:TerminateInstances.

5. Resource Control Policies (RCPs)

RCPs are a newer addition to AWS Organizations. Where SCPs restrict what identities in an account can do, RCPs restrict what can be done to resources in an account. They protect resources from unauthorized access regardless of who is calling.

6. Session policies

Passed as a parameter when you programmatically assume a role or federate a user. Session policies further restrict the temporary credentials. The effective permissions are the intersection of the role's identity policy, any permission boundary, and the session policy.

  • Session policies cannot grant more permissions than the role's policy.
  • They are useful for scoping down access on a per-session basis.

The anatomy of a policy document

Every IAM policy is JSON with a defined structure. Knowing every element cold is essential.

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Read",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "StringEquals": {
          "s3:prefix": "home/"
        }
      }
    }
  ]
}

Element-by-element breakdown

Version: Always use "2012-10-17". The only other valid value is "2008-10-17", which is legacy and does not support policy variables like ${aws:username}. If you omit Version, it defaults to 2008-10-17.

Statement: An array of one or more permission statements. Each statement is evaluated independently.

Sid (Statement ID): Optional. A human-readable identifier for the statement. Must be unique within the policy.

Effect: "Allow" or "Deny". There is no other option.

Principal: Who the policy applies to. Only used in resource-based policies and trust policies. Values can be:

Principal type Syntax
AWS account"AWS": "arn:aws:iam::123456789012:root"
IAM user"AWS": "arn:aws:iam::123456789012:user/Alice"
IAM role"AWS": "arn:aws:iam::123456789012:role/MyRole"
AWS service"Service": "ec2.amazonaws.com"
Everyone (anonymous)"*"
Federated user"Federated": "cognito-identity.amazonaws.com"

Action: The API operation(s) being allowed or denied. Wildcards are supported: "s3:*" means all S3 actions; "s3:Get*" means all S3 Get actions.

Resource: The ARN(s) of the resource(s) the statement applies to. Use "*" for all resources. Some actions (like s3:ListAllMyBuckets) require "*" as the resource.

Condition: Optional. Makes the statement apply only when specific conditions are met.

The "Not" variants

IAM supports negation elements that invert matching:

  • NotAction: Matches every action except those listed. Useful with Deny to block everything except a few actions.
  • NotResource: Matches every resource except those listed.
  • NotPrincipal: Matches every principal except those listed. AWS recommends avoiding NotPrincipal in favor of using Condition with aws:PrincipalArn and ArnNotEquals.

Exam trap: NotAction with Effect: Allow grants access to every action except the listed ones. This is dangerously broad. NotAction with Effect: Deny is the safer pattern.

Condition keys and operators

Conditions add granular control. The structure is:

"Condition": {
  "ConditionOperator": {
    "ConditionKey": "Value"
  }
}

How AWS evaluates policies - the decision engine

This is the single most important concept for debugging IAM and for the exam. The evaluation order determines whether a request is allowed or denied.

The core rules

  1. By default, all requests are implicitly denied. Nothing is allowed unless a policy explicitly says so.
  2. An explicit Allow overrides an implicit deny.
  3. An explicit Deny always wins. It overrides any and all Allow statements, no matter where they appear.

The full evaluation order

When a request arrives, AWS evaluates policies in this sequence:

  1. Explicit deny check - scan all applicable policies for a Deny that matches. If found, the request is denied immediately.
  2. SCPs - if the account is in an Organization, check SCPs.
  3. RCPs - if applicable, check resource control policies.
  4. Resource-based policies - if the resource has a resource-based policy that allows the request and names the principal directly, the request may be allowed even without an identity-based policy allowing it.
  5. Permission boundaries - if the identity has a permission boundary, check that it allows the action.
  6. Session policies - if this is a session created by AssumeRole with session policies, check that they allow the action.
  7. Identity-based policies - check the user's or role's attached policies for an Allow.
  8. Final implicit deny - if nothing explicitly allowed the action, it is denied.

Same-account vs. cross-account evaluation - critical difference

Same account: Identity-based policies and resource-based policies are evaluated together. If either one allows the action (and nothing explicitly denies it), the request succeeds.

Cross-account: Both sides must allow. The identity in Account A must have permission to perform the action, and the resource in Account B must have a resource-based policy (or a role assumption) that allows Account A's principal.

Exception - same-account resource-based policy with direct principal: If a resource-based policy in the same account grants access directly to an IAM user or session principal, that grant is not limited by permission boundaries or session policies.

The KMS exception

KMS key policies behave differently from most resource-based policies. For KMS, the key policy is mandatory. Even in the same account, even with an administrator identity policy allowing kms:*, if the key policy does not trust the principal or the account root, the request is denied.

Exam trigger: "Administrator has full IAM permissions but receives Access Denied on a KMS key" - answer: the key policy does not trust the principal.

How one AWS service accesses another

The dominant pattern in AWS is service-to-service access through roles.

Pattern: Service assumes a role

  1. You create a role.
  2. You attach a trust policy that allows the service to assume it.
  3. You attach a permissions policy that defines what the role can do.
  4. You associate the role with the service.
  5. The service receives temporary credentials from STS and uses them to call the target service.

Scenario 1 - EC2 accessing S3 (same account)

Trust policy on the role:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ec2.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}

Permissions policy on the role:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::my-bucket",
      "arn:aws:s3:::my-bucket/*"
    ]
  }]
}

Two resource ARNs are required: arn:aws:s3:::my-bucket covers bucket-level actions like ListBucket. arn:aws:s3:::my-bucket/* covers object-level actions like GetObject and PutObject.

S3 bucket policy: Not required in same-account access. The identity policy alone is sufficient.

How EC2 gets credentials: The instance contacts the Instance Metadata Service (IMDS) at 169.254.169.254. IMDS returns temporary credentials associated with the instance's role. The AWS SDK handles this automatically.

Scenario 2 - EC2 accessing S3 (cross-account)

The trust policy on Account A's role is identical to Scenario 1. What changes: Account B's bucket needs a bucket policy that explicitly allows Account A's role.

Bucket policy on Account B's S3 bucket:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "AWS": "arn:aws:iam::111111111111:role/EC2-S3-Role" },
    "Action": ["s3:GetObject", "s3:PutObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::account-b-bucket",
      "arn:aws:s3:::account-b-bucket/*"
    ]
  }]
}

Both sides must allow: Account A's role must have permissions to call S3 actions on the cross-account bucket, and Account B's bucket policy must trust Account A's role.

Scenario 3 - Lambda accessing DynamoDB (same account)

Same pattern as EC2-to-S3 with two changes: the trusted service is lambda.amazonaws.com, and every Lambda execution role should also include CloudWatch Logs permissions for function logging.

Permissions policy includes:

{
  "Effect": "Allow",
  "Action": ["logs:CreateLogGroup", "logs:CreateLogStream", "logs:PutLogEvents"],
  "Resource": "arn:aws:logs:*:*:*"
}

DynamoDB does not support resource-based policies, so the identity policy alone governs access within the same account.

Scenario 4 - Lambda accessing DynamoDB (cross-account)

Because DynamoDB does not support resource-based policies, you cannot use the bucket-policy pattern. Instead, you must use role assumption across accounts:

  1. Account A's Lambda execution role gets permission to call sts:AssumeRole on Account B's role.
  2. Account B creates a role with a trust policy naming Account A's Lambda execution role.
  3. Account B's role has a permissions policy granting DynamoDB actions.
  4. The Lambda function code must explicitly call sts:AssumeRole, receive temporary credentials, and use those credentials to create a DynamoDB client for Account B.

Cross-account summary table

Scenario Role in Account A Role in Account B Resource policy AssumeRole in code
EC2 -> S3 same accountYes (EC2 trust)NoOptionalNo
EC2 -> S3 cross-accountYes (EC2 trust)NoYes (mandatory)No
Lambda -> DynamoDB sameYes (Lambda trust)NoN/ANo
Lambda -> DynamoDB crossYes (Lambda trust)Yes (named role trust)N/AYes

The unifying pattern

  • Same account + resource policy support (S3, SQS, SNS, Lambda, KMS): identity policy is enough; resource policy optional (except KMS where key policy is always required).
  • Same account + no resource policy support (DynamoDB, EC2): identity policy is enough.
  • Cross-account + resource policy support (S3, SQS, SNS, Lambda): identity policy in Account A + resource policy in Account B.
  • Cross-account + no resource policy support (DynamoDB, EC2): role in Account A with sts:AssumeRole permission, role in Account B with trust for A's role, and explicit AssumeRole call in code.

SCP key rules

  • SCPs do not apply to the management account.
  • SCPs do not affect service-linked roles.
  • SCPs apply to all IAM users and roles in the affected accounts, including the root user.
  • SCPs are inherited from root to OU to account.
  • The effective SCP is the intersection of all SCPs from root down to the account.
  • AWS provides a default SCP called FullAWSAccess that allows everything.

SCP strategies

Deny list (recommended): Keep the default FullAWSAccess SCP and add explicit denies.

Allow list: Remove the default and create SCPs that explicitly allow only what you want. More secure but harder to maintain.

The permission layers - complete mental model

Layer What it does Grants permissions? Can be overridden?
SCPMaximum boundary for the account/OUNo - ceiling onlyOnly by removing the SCP
RCPMaximum boundary for resources in the accountNo - ceiling onlyOnly by removing the RCP
Permission boundaryMaximum boundary for a user or roleNo - ceiling onlyOnly by removing the boundary
Identity-based policyGrants permissions to the identityYesBy explicit deny from any layer
Resource-based policyGrants permissions from the resource sideYesBy explicit deny from any layer
Session policyFurther restricts a temporary sessionNo - ceiling onlyOnly by not passing it

The formula: Effective permissions = (Identity policy union Resource policy) intersect Permission boundary intersect SCP intersect RCP intersect Session policy, minus any explicit deny from any source.

Simplified rule: For an action to succeed, it must be allowed by at least one granting policy, and it must not be blocked by any ceiling policy, and it must not be explicitly denied anywhere.

IAM Identity Center (formerly AWS SSO)

For multi-account environments, IAM Identity Center is the recommended way to manage human access.

How it works

  1. Connect IAM Identity Center to an identity source.
  2. Create permission sets.
  3. Assign permission sets to users or groups for specific accounts.
  4. IAM Identity Center automatically creates IAM roles in those accounts with the correct trust and permissions.

Key benefits

  • Single sign-on
  • Centralized management
  • Temporary credentials
  • Multi-account access

Federation

Federation allows external identities to access AWS resources without creating IAM users.

SAML 2.0 federation

Used for enterprise federation with Active Directory, ADFS, Okta, and similar systems. The identity provider sends a SAML assertion to AWS, which maps it to a role via AssumeRoleWithSAML.

Web identity federation

Used for mobile or web users authenticating with Google, Facebook, Amazon, or any OIDC provider. Uses AssumeRoleWithWebIdentity. AWS recommends using Amazon Cognito instead of calling STS directly.

Custom identity broker

For legacy systems that do not support SAML. Your code authenticates users against your identity store, then calls GetFederationToken or AssumeRole.

Amazon Cognito

Cognito handles authentication and authorization for the users of your applications (not your workforce - that is IAM Identity Center).

Cognito User Pools

  • Handle authentication: sign-up, sign-in, password recovery, MFA.
  • Return JSON Web Tokens (JWTs).
  • Support social sign-in, SAML, and OIDC providers.

Cognito Identity Pools

  • Handle authorization into AWS: exchange a token for temporary IAM credentials.
  • Support authenticated and unauthenticated identities.
  • Map users to IAM roles.

The combined flow

  1. User signs in via Cognito User Pool and receives JWT.
  2. JWT is sent to Cognito Identity Pool and exchanged for temporary IAM credentials.
  3. User uses temporary credentials to access AWS resources.

IMDSv1 vs. IMDSv2

EC2 instances retrieve their role credentials from the Instance Metadata Service (IMDS) at http://169.254.169.254.

IMDSv1 (legacy)

A simple HTTP GET request to the metadata endpoint returns data. Any process on the instance can access it. This is vulnerable to Server-Side Request Forgery (SSRF) attacks.

IMDSv2 (recommended)

Session-oriented. Two-step process:

  1. PUT request to obtain a session token.
  2. Use the session token in the header of subsequent GET requests.

Enforcing IMDSv2: Set HttpTokens to required on the instance's metadata options.

IAM best practices

  1. Lock away the root user access keys.
  2. Require MFA for privileged actions and for all human users.
  3. Grant least privilege.
  4. Use roles instead of access keys for applications and services.
  5. Use IAM Identity Center for human workforce access.
  6. Rotate credentials regularly.
  7. Use policy conditions for extra security.
  8. Use permission boundaries when delegating IAM administration.
  9. Monitor with CloudTrail, Access Analyzer, and Config rules.
  10. Do not share credentials.
  11. Use customer managed policies over inline policies.
  12. Remove unused users, roles, and credentials.

Service-linked roles

A service-linked role is an IAM role that is linked directly to an AWS service. The service creates and manages the role, including its trust policy and permissions. You cannot modify the permissions of a service-linked role.

Key properties:

  • Created by the service automatically or manually via IAM.
  • The trust policy allows only the linked service to assume the role.
  • You can delete a service-linked role only after confirming the service is no longer using it.
  • SCPs do not restrict service-linked roles.

iam:PassRole - the hidden gatekeeper

When you attach a role to an AWS service, you are performing an action called iam:PassRole. This is a permission check, not an API call - it verifies that the user passing the role has permission to do so.

Policy example:

{
  "Effect": "Allow",
  "Action": "iam:PassRole",
  "Resource": "arn:aws:iam::123456789012:role/LambdaExecutionRole",
  "Condition": {
    "StringEquals": {
      "iam:PassedToService": "lambda.amazonaws.com"
    }
  }
}

This says: "You can pass the LambdaExecutionRole role, but only to the Lambda service."

MFA Delete for S3

S3 versioning supports MFA Delete, requiring MFA to delete object versions or change the versioning state of a bucket. This can only be enabled by the root user via the CLI.

Resource-based policies - deep dive

S3 bucket policies

Bucket policies are the most common resource-based policy on the exam. Key scenarios:

  • Public access: A bucket policy with Principal: "*" grants public access (subject to Block Public Access settings).
  • Cross-account access: A bucket policy naming another account's role allows that role to access the bucket without the role needing to call AssumeRole.
  • VPC endpoint restriction: A bucket policy with a condition on aws:SourceVpce restricts access to a specific VPC endpoint.
  • Encryption enforcement: A bucket policy that denies s3:PutObject unless the encryption header is present.

S3 Block Public Access

Account-level and bucket-level settings that override any policy that would grant public access.

Four settings:

  • BlockPublicAcls
  • IgnorePublicAcls
  • BlockPublicPolicy
  • RestrictPublicBuckets

Cross-account access patterns - complete reference

Pattern 1: Resource-based policy (when supported)

Account B's resource policy names Account A's principal. No role assumption needed.

Used for: S3, SQS, SNS, Lambda, KMS, Secrets Manager.

Pattern 2: Role assumption

Account A's role gets sts:AssumeRole permission for Account B's role. Account B's role trusts Account A's role.

Used for: Any service, but required for services without resource-based policies such as DynamoDB and EC2.

Pattern 3: AWS Organizations with aws:PrincipalOrgID

{
  "Condition": {
    "StringEquals": {
      "aws:PrincipalOrgID": "o-xxxxxxxxxxx"
    }
  }
}

Advantage: Automatically covers new accounts added to the Organization.

The cross-account handshake (Principle 5)

  1. Account A's identity needs an IAM policy allowing sts:AssumeRole on Account B's role ARN.
  2. Account B's role needs a trust policy naming Account A's identity as a trusted principal.

Neither side alone is sufficient.

Confusing deputies and external IDs

The confused deputy problem

When a third-party AWS service assumes a role in your account, a malicious customer of that same service could trick it into accessing your resources by providing your role ARN. The third-party service acts as a "confused deputy."

The solution: External ID

{
  "Condition": {
    "StringEquals": {
      "sts:ExternalId": "unique-value-from-partner"
    }
  }
}

The third-party service must supply this external ID when calling AssumeRole.

Common exam traps and edge cases

  1. Explicit deny cannot be overridden.
  2. SCPs do not grant permissions.
  3. Resource ARN specificity for S3.
  4. KMS key policy is mandatory.
  5. Groups are not identities.
  6. Permission boundaries limit, they do not grant.
  7. Trust policies live on the role, not the caller.
  8. EC2 Resource: "*".
  9. NotAction with Allow is dangerously broad.
  10. STS endpoints and Region.

Final mental model

Every IAM scenario in the exam can be broken down with this sequence:

  1. Who is the actor? (User, role, service, federated identity)
  2. What is the target? (Which service and resource)
  3. Is a role involved? (Almost always yes for service access)
  4. Is the actor trusted to assume the role? (Trust policy check)
  5. Does the role have permission for the action? (Permissions policy check)
  6. Is this cross-account? (If yes, check both sides)
  7. Are there ceiling policies? (SCP, permission boundary, session policy)
  8. Is there any explicit deny? (If yes, denied)

If you can answer these eight questions for any scenario, you can answer any IAM question on the SAA-C03.

AWS identity and access management

AWS IAM, explained like an article instead of a cheat sheet

IAM is the system that decides who can do what in AWS. If S3, EC2, Lambda, DynamoDB, and your own engineers are all actors inside AWS, IAM is the rulebook that controls access between them.

For personal revision Medium-style reading layout Focused on concepts and examples

What is IAM, really?

IAM stands for Identity and Access Management. In practical terms, it answers two questions every time something tries to act in AWS:

  1. Who is making this request?
  2. Is this request allowed?

That “something” might be a human user opening the AWS console, an EC2 instance trying to read from S3, a Lambda function writing to DynamoDB, or one AWS account trying to access a resource in another account.

A good way to think about IAM is this: AWS services do not trust each other automatically. Access must be granted explicitly.

The core IAM objects: users, roles, and policies

Users

A user is a long-term identity, usually representing a human or sometimes an older application that still uses access keys. Users can have a console password, access keys, or both.

Example

If you log into the AWS console with your own email-backed account and click around S3, EC2, or IAM itself, that activity is typically happening as an IAM user or a federated identity mapped into IAM permissions.

Groups

A group is just a way to collect users and attach the same permissions to all of them. Groups are useful for human administration, but they are not identities by themselves. You cannot sign in “as a group.”

Roles

A role is a temporary identity that can be assumed. This is the object that matters most for modern AWS architecture because services like EC2 and Lambda normally use roles rather than hard-coded credentials.

Roles do not have long-term passwords or access keys attached in the same way users do. Instead, AWS issues temporary credentials through STS when a trusted principal assumes the role.

Example

If an EC2 instance needs to read from an S3 bucket, the recommended setup is not to store AWS keys on the server. Instead, attach a role to the instance, let EC2 assume that role, and let AWS provide temporary credentials automatically.

Policies

A policy is the JSON document that defines permissions. Policies answer questions like:

  • Can this identity call s3:GetObject?
  • Can this Lambda function write to DynamoDB?
  • Can this principal assume a role?

In IAM, policies are the language of authorization. They can allow or deny actions, and AWS evaluates them together to reach a decision.

Why roles matter more than users in cloud architecture

In a real AWS system, most access is not human-to-service. It is service-to-service. That means the most common IAM design question is not “which user should get this permission?” but rather:

Which service or workload should assume a role, and what should that role be allowed to do?

That is why roles are central. They are the clean way to let workloads act inside AWS without embedding secrets.

How one AWS service accesses another

The most common pattern is: a service assumes a role, then uses the permissions on that role to call another service.

Example: EC2 accesses S3

The flow looks like this:

  1. You create a role, for example S3AccessRole.
  2. You attach a trust policy that allows EC2 to assume it.
  3. You attach a permissions policy that allows S3 actions like s3:GetObject.
  4. You attach the role to the EC2 instance.
  5. The EC2 instance receives temporary credentials and uses them to call S3.
Key distinction

The trust policy answers: who is allowed to assume this role? The permissions policy answers: what can the role do once it has been assumed?

Example trust policy

This trust policy lives on the role itself. It says EC2 is allowed to assume the role. 

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

Example permissions policy

This permissions policy also attaches to the role, but it does a different job. It defines what the role can do after EC2 has assumed it. 

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    }
  ]
}

Example: Lambda writes to DynamoDB

This is the same pattern with a different trusted service. The role’s trust policy allows lambda.amazonaws.com, and the permissions policy allows the DynamoDB actions the function needs, such as dynamodb:PutItem.

What is a resource policy?

A resource policy is a policy attached to the resource itself instead of to the caller. That means the bucket, queue, or key is stating who is allowed to access it.

Not every AWS service supports resource policies, but when they do exist, they are especially important for cross-account access because the target resource can explicitly trust a principal from somewhere else.

Simple distinction

Identity policy: attached to a user or role, describing what that identity is allowed to do. Resource policy: attached to the target resource, describing who may access that resource.

Example: S3 bucket policy

This bucket policy is a resource policy. It lives on the S3 bucket and allows a specific role to read objects from that bucket. 

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:role/S3AccessRole"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-app-bucket/*"
    }
  ]
}

In same-account access, you often do not need a bucket policy if the role’s identity policy already grants access. In cross-account access, the bucket policy is often the missing piece that makes the whole request succeed.

Core principles first

Before the scenarios, lock in these five principles. Every scenario below is just an application of them.

  • Principle 1 — The role is always the middleman. No AWS service like EC2 or Lambda ever holds permissions directly. It assumes a role, and the role holds the permissions. Think of the role as a uniform that grants access and the service puts it on temporarily.
  • Principle 2 — Every role has exactly two policies. A trust policy defines who is allowed to assume the role, and a permissions policy defines what the role can do once assumed. The trust policy is the lock. The permissions policy is the key to what is inside.
  • Principle 3 — Trust policies always live on the role, never on the caller. EC2 does not say “I trust this role.” The role says “I trust EC2.” This is the most common source of confusion and one of the most tested IAM ideas.
  • Principle 4 — Same account vs cross-account changes everything about resource policies. For same-account access, an identity policy alone is sufficient. For cross-account access, both the identity side and the resource side must explicitly allow. A missing bucket policy in the target account is an instant Access Denied.
  • Principle 5 — Cross-account role assumption requires a handshake from both sides. Account A needs permission to call sts:AssumeRole on Account B’s role, and Account B’s role needs a trust policy that names Account A’s role as trusted. Neither side alone is enough.

The Principal field — know these cold

When you write a trust policy, the Principal field is what changes between scenarios. These are the values you use most often.

Who is assuming the role Principal value
EC2 service "Service": "ec2.amazonaws.com"
Lambda service "Service": "lambda.amazonaws.com"
A specific IAM role (cross-account) "AWS": "arn:aws:iam::ACCOUNT_ID:role/ROLE_NAME"
A specific IAM user "AWS": "arn:aws:iam::ACCOUNT_ID:user/USERNAME"
All principals in an AWS organization "AWS": "*" with condition aws:PrincipalOrgID

Scenarios reference

Scenario 1 — EC2 accessing S3 in the same account

Where you create the role: Account A, where both EC2 and S3 live. What you create: one IAM role attached to EC2 through an instance profile.

Trust policy — on the role in Account A:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "ec2.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}

This says: “I, this role, trust the EC2 service to assume me.” EC2 automatically assumes this role when the instance starts. The instance profile is the container that connects the role to the EC2 instance.

Permissions policy — on the same role in Account A:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "s3:GetObject",
      "s3:PutObject",
      "s3:DeleteObject",
      "s3:ListBucket"
    ],
    "Resource": [
      "arn:aws:s3:::my-bucket",
      "arn:aws:s3:::my-bucket/*"
    ]
  }]
}

The two resource lines matter: arn:aws:s3:::my-bucket covers ListBucket, and arn:aws:s3:::my-bucket/* covers object-level actions. Missing one of them causes partial failures.

S3 bucket policy: not required. Same-account access works if the identity policy allows it. The app gets temporary credentials automatically from the instance metadata endpoint.

Scenario 2 — EC2 accessing S3 in a different AWS account

Where you create things: a role in Account A where EC2 lives, and a bucket policy in Account B where S3 lives. No role is needed in Account B because S3 supports resource-based policies directly.

Trust policy — on the role in Account A:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "ec2.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}

This trust policy is identical to the same-account scenario. The caller is still EC2. What changes is the resource location and the need for a bucket policy in the target account.

Permissions policy — on the role in Account A:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "s3:GetObject",
      "s3:PutObject",
      "s3:ListBucket"
    ],
    "Resource": [
      "arn:aws:s3:::account-b-bucket",
      "arn:aws:s3:::account-b-bucket/*"
    ]
  }]
}

Bucket policy — on the S3 bucket in Account B:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::111111111111:role/EC2-S3-Role"
    },
    "Action": [
      "s3:GetObject",
      "s3:PutObject",
      "s3:ListBucket"
    ],
    "Resource": [
      "arn:aws:s3:::account-b-bucket",
      "arn:aws:s3:::account-b-bucket/*"
    ]
  }]
}

This is Account B explicitly opening the door to Account A’s specific role. Without this, you get Access Denied regardless of what Account A’s identity policy says.

Scenario 3 — Lambda accessing DynamoDB in the same account

Where you create the role: Account A, same account as Lambda and DynamoDB. What you create: one execution role attached to the Lambda function.

Trust policy — on the execution role in Account A:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "lambda.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}

Permissions policy — on the execution role in Account A:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:GetItem",
        "dynamodb:PutItem",
        "dynamodb:UpdateItem",
        "dynamodb:DeleteItem",
        "dynamodb:Query",
        "dynamodb:Scan"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:111111111111:table/Orders"
    },
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:*:*:*"
    }
  ]
}

The only trust-policy difference from EC2 is lambda.amazonaws.com instead of ec2.amazonaws.com. Also note the CloudWatch Logs permissions, which every Lambda execution role should have.

Scenario 4 — Lambda accessing DynamoDB in a different account

Where you create things: an execution role in Account A and a cross-account role in Account B. DynamoDB is accessed via role assumption, not a bucket-style resource policy.

Trust policy — on LambdaExecRole in Account A:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "lambda.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}

Permissions policy — on LambdaExecRole in Account A:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::222222222222:role/DynamoDBCrossAcctRole"
    },
    {
      "Effect": "Allow",
      "Action": ["logs:CreateLogGroup","logs:CreateLogStream","logs:PutLogEvents"],
      "Resource": "arn:aws:logs:*:*:*"
    }
  ]
}

Trust policy — on DynamoDBCrossAcctRole in Account B:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::111111111111:role/LambdaExecRole"
    },
    "Action": "sts:AssumeRole"
  }]
}

Permissions policy — on DynamoDBCrossAcctRole in Account B:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "dynamodb:GetItem",
      "dynamodb:PutItem",
      "dynamodb:Query"
    ],
    "Resource": "arn:aws:dynamodb:us-east-1:222222222222:table/SharedOrders"
  }]
}

Here the Lambda code must call sts:AssumeRole explicitly, receive temporary credentials, and then use those credentials to create a DynamoDB client for Account B.

Scenario 5 — Lambda accessing EC2 in a different account

This is structurally identical to Scenario 4. The only real difference is that Account B’s role grants EC2 actions instead of DynamoDB actions.

Trust policy — on LambdaExecRole in Account A:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Service": "lambda.amazonaws.com"
    },
    "Action": "sts:AssumeRole"
  }]
}

Permissions policy — on LambdaExecRole in Account A:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::222222222222:role/EC2ManagementRole"
    },
    {
      "Effect": "Allow",
      "Action": ["logs:CreateLogGroup","logs:CreateLogStream","logs:PutLogEvents"],
      "Resource": "arn:aws:logs:*:*:*"
    }
  ]
}

Trust policy — on EC2ManagementRole in Account B:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::111111111111:role/LambdaExecRole"
    },
    "Action": "sts:AssumeRole"
  }]
}

Permissions policy — on EC2ManagementRole in Account B:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "ec2:StopInstances",
      "ec2:StartInstances",
      "ec2:DescribeInstances",
      "ec2:RebootInstances"
    ],
    "Resource": "*"
  }]
}

EC2 often requires Resource: "*" for many API calls. Least privilege usually comes from conditions, such as restricting by instance tags.

Comparison summary

Scenario Role in Account A Role in Account B Resource policy sts:AssumeRole in code
EC2 → S3 same account Yes — EC2 trust No Optional No
EC2 → S3 cross account Yes — EC2 trust No Yes — mandatory No
Lambda → DynamoDB same Yes — Lambda trust No Not supported No
Lambda → DynamoDB cross Yes — Lambda trust Yes — named role ARN trust Not applicable Yes
Lambda → EC2 cross Yes — Lambda trust Yes — named role ARN trust Not applicable Yes

The unifying pattern is simple:

  • Same account + service with resource policy support like S3: identity policy is enough, resource policy optional.
  • Same account + service without resource policy support like DynamoDB or EC2: identity policy is enough.
  • Cross account + service with resource policy support like S3: identity policy in A plus resource policy in B.
  • Cross account + service without resource policy support like DynamoDB or EC2: role in A with sts:AssumeRole, role in B with trust for A’s role, and an explicit AssumeRole call in code.

Why trust policies confuse people

A common mistake is assuming the calling service should define the trust. It does not. The trust policy lives on the role being assumed.

Useful mental model

Think of a role as a uniform hanging in a cabinet. The trust policy is the lock on that cabinet. The entity trying to wear the uniform does not define the lock. The role does.

What is AWS Organizations?

AWS Organizations is the service used to manage multiple AWS accounts together as one governed structure. Instead of treating every AWS account as an isolated island, Organizations lets you group accounts under one management layer.

This is useful when a company has separate accounts for production, staging, security, analytics, or different teams. Rather than configuring everything separately in each account, Organizations lets you apply broad governance rules at a higher level.

Example

A company may have one organization with a root, then organizational units such as Production, Sandbox, and Security. Each OU can contain multiple AWS accounts.

The most important IAM-related concept inside AWS Organizations is the Service Control Policy, or SCP.

The different layers of permissions in AWS

One reason IAM feels confusing is that AWS permissions do not come from one single place. Access can be shaped by multiple layers at once. The clean way to think about them is from outer boundary to inner permission.

1. Service Control Policies (SCPs)

An SCP is part of AWS Organizations. It does not grant permissions by itself. Instead, it sets the maximum boundary for what accounts inside the organization are allowed to do.

If an SCP denies an action, then no IAM user or role inside that account can bypass it, even if their own IAM policy says Allow.

Important rule

SCPs are guardrails, not permission grants. They define what is possible at the account or OU level.

Example

If an SCP denies ec2:TerminateInstances in the Production OU, then even an administrator role in a production account cannot terminate an instance unless the SCP is changed.

2. Permission boundaries

A permission boundary is attached to a user or role and limits the maximum permissions that identity can ever have. Like SCPs, it is a ceiling, not a direct grant.

This is useful when you want to delegate IAM creation to teams, but still prevent them from creating overly powerful roles.

3. Identity-based policies

These are the regular IAM policies attached to users, groups, or roles. This is the layer most people think of first when they hear “IAM policy.”

Identity policies are what usually grant day-to-day permissions like s3:GetObject, dynamodb:PutItem, or ec2:DescribeInstances.

4. Resource-based policies

These live on the resource itself, such as an S3 bucket policy, an SQS queue policy, or a KMS key policy. They matter a lot in cross-account access because the target resource can explicitly say who is trusted.

5. Session policies and temporary credentials

When a role is assumed, AWS can also apply session-level restrictions. These matter less for basic understanding, but they are another layer that can make effective permissions narrower than expected.

How these layers combine

The practical rule is: effective permissions are the intersection of what all the limiting layers allow, minus anything explicitly denied.

  • SCP says what the account can ever do.
  • Permission boundary says what the identity can ever do.
  • Identity policy says what the identity is trying to do.
  • Resource policy may also need to allow access.
  • Any explicit deny at any layer can stop the request.
Concrete example

Imagine a role has an IAM policy allowing s3:GetObject. If the organization SCP denies all S3 actions, the request still fails. If the SCP allows S3 but the bucket policy does not trust that role in a cross-account case, the request still fails. The final answer is never based on just one policy document.

Five advanced domains that move you from concepts to exam readiness

To move from conceptual understanding to SAA-C03 readiness, you need to know a few IAM-adjacent topics that show up in architecture scenarios. They are not always the headline of the question, but they often decide which answer is correct.

1. The metadata security layer: IMDSv1 vs IMDSv2

EC2 instances retrieve temporary credentials from the Instance Metadata Service at 169.254.169.254. That is powerful, but it also creates a security question: how easy is it for a process on the server to reach that endpoint?

IMDSv1 uses a simple request-response model. A process can run something like curl http://169.254.169.254/latest/meta-data/ and start retrieving information directly.

IMDSv2 is session-oriented. The caller must first do a PUT request to obtain a temporary session token, then supply that token on subsequent metadata requests.

Exam trigger

If a question mentions SSRF attacks, hardening instance metadata, or making metadata access more secure, the intended answer is usually to enforce IMDSv2 by setting HttpTokens to required.

2. IAM Identity Center and centralized governance

In a multi-account AWS environment, you usually do not manage separate IAM users inside every account. Instead, you use IAM Identity Center to centralize workforce access.

IAM Identity Center connects AWS to an identity source such as Okta, Azure AD, or Active Directory using federation standards like SAML 2.0. You define permission sets centrally, assign them to users or groups for specific accounts, and AWS creates the matching IAM roles in those member accounts automatically.

Exam trigger

If the scenario says “centrally manage access to multiple AWS accounts” or “let employees use existing corporate credentials to access AWS,” think IAM Identity Center rather than manually creating IAM users in each account.

3. Advanced auditing and troubleshooting tools

SAA-C03 expects you to know not only how to grant access, but also how to reduce, validate, and audit it.

  • IAM Access Advisor: shows when a service was last used by an identity. This helps identify unused permissions.
  • IAM Access Analyzer: inspects resource-based policies and alerts on unintended public or cross-account access.
  • IAM Policy Simulator: lets you test whether a policy would allow or deny an action before relying on it in production.
  • Credential Report: generates an account-wide CSV showing password age, access key age, and MFA status.
Least-privilege angle

If the question is about identifying unused permissions, Access Advisor is the clue. If the question is about detecting public or external exposure from a bucket or KMS key, Access Analyzer is the clue.

4. The KMS resource-policy exception

KMS is a common exam trap because it behaves differently from the “same-account access usually works with identity policy alone” pattern.

With S3, a same-account IAM user with enough permissions can often access the bucket even if the bucket has no bucket policy. With KMS, the key policy is mandatory. If the key policy does not trust the principal or the account root appropriately, even an administrator can receive Access Denied when trying to use the key.

Exam trigger

If an administrator appears to have full IAM permissions but still cannot decrypt data or use a KMS key, the missing piece is often the key policy.

5. Amazon Cognito and external identity

IAM is mainly for your workforce, workloads, and AWS-managed service access. Amazon Cognito is for the users of the applications you build.

Cognito User Pools handle authentication: sign-up, sign-in, and token issuance. They return JWTs. Cognito Identity Pools handle authorization into AWS: they take an authenticated identity and exchange it for temporary IAM credentials.

Exam trigger

If a question asks how mobile or web users can sign in with Google or Facebook and then upload files directly to S3, the expected pattern is Cognito for identity plus temporary IAM credentials for AWS access.

What changes in cross-account access

Cross-account access adds one more layer of thinking because now you have two security boundaries: the source account and the target account.

Suppose an EC2 instance in Account A needs to access an S3 bucket in Account B. In that case, it is not enough for Account A to “want” access. Account B must also explicitly allow it.

Cross-account example

Account A may use a role with permission to access the bucket, but the bucket policy in Account B must still allow that principal. Cross-account access is usually a handshake, not a one-sided permission grant.

This is one of the most important IAM habits to build: whenever the request crosses accounts, ask yourself which side is granting access and whether both sides of the path are configured.

How AWS decides: the basic evaluation idea

IAM can feel complex because many policy types can be involved at once: identity policies, resource policies, permission boundaries, and even organization-level SCPs.

But the core rule is simple:

  • Explicit deny wins.
  • If nothing allows the action, the result is implicit deny.

Most IAM debugging comes down to finding where the deny is happening or where the expected allow is missing. In other words, IAM decisions are not only about what one role says it can do, but also about what the surrounding governance layers permit.

A practical mental model for revising IAM quickly

If you want to remember IAM without memorizing scattered facts, use this sequence whenever you look at a scenario:

  1. Identify the actor. Who is making the request?
  2. Identify the target. Which service or resource is being accessed?
  3. Check whether a role is involved. Most service access uses roles.
  4. Check trust. Is the actor allowed to assume the role?
  5. Check permissions. Once assumed, can that role call the needed actions?
  6. If cross-account, check both sides.
Short version

IAM is usually not hard because the JSON is hard. It is hard because people mix up identity, trust, permissions, and resource ownership.

Final takeaway

IAM is the control layer behind almost every meaningful AWS interaction. Users are long-term identities. Roles are temporary identities. Policies define permissions. Trust policies decide who can assume a role. And once you understand how one service assumes a role to reach another service, most AWS IAM questions start to look like variations of the same pattern.

That is the level you want for revision: not isolated facts, but one reusable mental model.

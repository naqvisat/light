What is RDS?

Amazon RDS (Relational Database Service) is a fully managed relational database service. AWS handles provisioning, patching, backups, recovery, and failover so you can focus on the application and the database design.

What "managed" means:

  • AWS manages OS patching, database engine patching, automated backups, Multi-AZ replication, storage scaling, and monitoring.
  • You manage schema design, query optimization, index tuning, and application-level security.
  • You cannot SSH into RDS instances. If you need full OS access, run your database on EC2 instead.

What is Aurora?

Amazon Aurora is a MySQL- and PostgreSQL-compatible relational database built by AWS for cloud-native performance and availability.

Why Aurora exists: Aurora delivers up to 5x the throughput of standard MySQL and 3x the throughput of standard PostgreSQL on RDS, often at a better performance-per-dollar point for larger workloads.

Aurora storage - 6 copies across 3 AZs

  • Writes require 4 out of 6 copies to acknowledge.
  • Reads require 3 out of 6 copies.
  • Aurora can tolerate the loss of 2 copies for writes and 3 copies for reads.
  • Storage is self-healing and continuously backed up to S3.

Multi-AZ deployments - high availability

Multi-AZ is about availability and failover, not read scaling.

RDS Multi-AZ

PropertyDetail
How it worksAWS maintains a synchronous standby replica in another AZ
StandbyClassic standby is not readable
FailoverAutomatic; DNS endpoint flips to standby in about 1-2 minutes
TriggerAZ outage, instance failure, storage failure, patching, or manual failover
EndpointSingle DNS endpoint that stays the same
CostYou pay for both primary and standby

RDS Multi-AZ DB Cluster is the newer model for MySQL and PostgreSQL: one writer plus two readable standbys across 3 AZs, with reader and writer endpoints and failover often under 35 seconds.

Aurora Multi-AZ

Aurora storage is already Multi-AZ. For compute-level HA, you place one or more Aurora Replicas in other AZs. If the writer fails, Aurora promotes a replica automatically, typically in under 30 seconds.

Exam trap: "Multi-AZ standby cannot be used for reads" applies to classic RDS Multi-AZ only. Aurora Replicas and the newer RDS Multi-AZ DB Cluster standbys can serve reads.

RDS Storage Auto Scaling

RDS automatically increases storage when free space falls below 10%, that condition lasts at least 5 minutes, and at least 6 hours have passed since the last storage modification.

Exam trigger: "RDS running out of storage" means enable Storage Auto Scaling.

Aurora storage

  • No storage provisioning required.
  • Grows automatically in 10 GiB increments.
  • Maximum size is 128 TiB.
  • You pay only for storage used.
  • Continuously backed up to S3.

Manual snapshots

  • Retained until you delete them.
  • Can be copied cross-Region.
  • Can be shared across accounts.

Aurora backtrack

Aurora MySQL can rewind the existing cluster to a prior point in time without restoring a new cluster. The backtrack window can be as large as 72 hours.

Exam trigger: "Quickly undo a bad database change without restoring to a new instance" means Aurora Backtrack.

Aurora clone

Aurora Clone creates a fast copy-on-write clone for testing or analytics. It is nearly instant and initially shares storage with the source until pages change.

Encrypting an existing unencrypted database

  1. Take a snapshot of the unencrypted database.
  2. Copy the snapshot and enable encryption during the copy.
  3. Restore a new encrypted DB instance from that snapshot.
  4. Switch the application to the new instance.

In transit

Use SSL/TLS for client connections. You can enforce SSL with parameter groups such as rds.force_ssl=1 for PostgreSQL or require_secure_transport=ON for MySQL.

RDS Proxy

RDS Proxy is a fully managed, highly available database proxy that sits between your application and your RDS or Aurora database.

What it solves

  • Connection pooling to protect the database from connection storms
  • Faster failover by reconnecting to the new target automatically
  • Lambda compatibility for spiky, short-lived connections

1 ACU is roughly equivalent to 2 GiB of memory plus proportional CPU and networking.

Aurora Serverless v1

  • Scales in larger steps and can pause during scaling.
  • Can scale to zero, which v2 cannot.
  • Has a more limited feature set than v2.

Exam trigger: unpredictable workload means Aurora Serverless v2. Truly intermittent development database that should cost nothing while idle points to Aurora Serverless v1.

Aurora Global Database

Aurora Global Database provides cross-Region replication with one primary Region and up to 5 secondary Regions.

  • Secondary Regions are read-only until promoted.
  • Replication lag is typically under 1 second.
  • Each secondary Region can have up to 16 Aurora Replicas.
  • Unplanned failover RPO is typically under 1 second, with RTO typically under 1 minute.

Exam trigger: "Cross-Region DR with under 1 second RPO" means Aurora Global Database.

IAM database authentication

RDS and Aurora support IAM-based authentication for MySQL and PostgreSQL.

  1. Enable IAM authentication on the DB instance.
  2. Create a database user mapped to IAM.
  3. The application obtains a token from IAM, valid for 15 minutes.
  4. The token is used instead of a database password.

Exam trigger: "Authenticate to RDS without managing database passwords" means IAM database authentication.

Security

Network security

  • Best practice is to run RDS in private subnets.
  • Use security groups to control who can connect.
  • Use a DB subnet group spanning multiple AZs.
  • Public accessibility is possible but usually a bad idea.

Parameter groups and option groups

Parameter groups hold engine settings like max_connections. Option groups enable engine-specific features such as Oracle TDE or SQL Server native backup.

Monitoring

CloudWatch metrics

Key metrics include CPUUtilization, FreeableMemory, ReadIOPS, WriteIOPS, DatabaseConnections, FreeStorageSpace, and ReplicaLag.

Enhanced Monitoring

Enhanced Monitoring provides OS-level metrics down to 1-second granularity.

Performance Insights

Performance Insights highlights the top SQL queries, waits, and sources of load. If the question asks which tool identifies slow queries or top SQL, the answer is Performance Insights.

Database migration

AWS Database Migration Service (DMS)

  • Supports homogeneous and heterogeneous migrations.
  • Supports continuous replication for low-downtime cutovers.
  • Use AWS Schema Conversion Tool (SCT) first for heterogeneous migrations.

Exam trigger: "Migrate on-premises Oracle to Aurora PostgreSQL with minimal downtime" means SCT + DMS.

Common exam traps and edge cases

  1. Classic RDS Multi-AZ standby is not for reads.
  2. Read replicas do not provide automatic failover.
  3. You cannot encrypt an existing unencrypted RDS instance in place.
  4. Encrypted replicas require an encrypted source.
  5. Aurora storage is automatic.
  6. Aurora failover is faster than standard RDS failover.
  7. RDS does not auto-scale compute.
  8. Stopped instances auto-start after 7 days.
  9. Aurora Global Database is not the same as standard cross-Region read replicas.
  10. RDS Proxy is VPC-only.
  11. Backtrack rewinds the existing Aurora cluster.
  12. DMS keeps the source database operational during migration.
  13. Aurora I/O-Optimized matters for I/O-heavy workloads.

Final mental model

Every RDS/Aurora exam question can be broken down with this sequence:

  1. Which engine? MySQL/PostgreSQL may point to Aurora; Oracle, SQL Server, and MariaDB stay on standard RDS.
  2. What availability is needed? Multi-AZ for same-Region HA; Aurora Global Database or cross-Region replicas for DR.
  3. What read scaling is needed? Use read replicas, Aurora Replicas, or caching.
  4. What is the workload pattern? Predictable means provisioned instances; unpredictable may mean Aurora Serverless v2.
  5. Is encryption required? Enable at creation; existing unencrypted databases require snapshot-copy-restore.
  6. What about connections? Spiky Lambda-style traffic points to RDS Proxy.
  7. What about cost? Reserved Instances, right-sizing, stopping dev/test, and Aurora I/O-Optimized all matter.
  8. Migration? DMS for data movement; SCT plus DMS for heterogeneous migrations.

If you can answer these eight questions for any scenario, you can answer any RDS/Aurora question on the SAA-C03.

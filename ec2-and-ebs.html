Part 1: EC2 - Elastic Compute Cloud

What is EC2?

EC2 provides resizable virtual servers (instances) in the cloud. You choose the operating system, instance type, and pay for what you use.

Key facts:

  • EC2 instances run in a specific Availability Zone within a Region.
  • Each instance runs on a physical host in an AWS data center, powered by the Nitro System.
  • Instances can be stopped, started, rebooted, or terminated.
  • When stopped, you are not charged for compute but still charged for EBS and Elastic IPs.
  • When terminated, the instance is deleted. Root EBS volumes are deleted by default.

Instance families

EC2 instance types follow a naming convention: m5.xlarge = family, generation, size.

T instances - burstable performance

T3/T3a/T4g instances use a CPU credit system.

  • Credits are earned when CPU usage is below baseline.
  • Credits are consumed when CPU bursts above baseline.
  • Unlimited mode can burst beyond earned credits and charge for surplus usage.
  • Standard mode throttles to baseline when credits are exhausted.

Graviton (ARM) instances

Instance types ending in g use AWS Graviton processors.

  • Up to 40% better price-performance than equivalent x86 instances.
  • Compatible with most Linux workloads.
  • Not compatible with Windows or x86-only binaries.

  • Up to 90% discount vs On-Demand.
  • AWS can reclaim with a 2-minute notification.

  • Diversify across types and AZs to reduce interruption risk.

AMIs (Amazon Machine Images)

An AMI is a template for launching instances. It includes the root volume, launch permissions, and block device mapping.

  • AMIs are Region-specific.
  • AMIs can be public, private, or shared.
  • Custom AMIs speed up boot time by pre-installing software.
  • You can create an AMI from a running or stopped instance.
  • AMIs can be backed by EBS or Instance Store.

User Data and Metadata

User Data

A script that runs once at first boot when an instance launches.

  • Limited to 16 KB.
  • Runs as the root user.
  • Available at http://169.254.169.254/latest/user-data.

Instance Metadata (IMDS)

Information about the instance available at http://169.254.169.254/latest/meta-data/.

Use IMDSv2 and enforce with HttpTokens=required.

EBS (Elastic Block Store)

EBS provides persistent block storage volumes for EC2 instances.

Volume types

sc1Cold HDD250250 MB/sInfrequent access

Key rules:

  • EBS volumes are AZ-specific.
  • Move to another AZ by snapshotting and restoring in target AZ.
  • Move to another Region by copying the snapshot.
  • Multi-Attach: io1/io2 only, up to 16 instances, same AZ only.
  • EBS can be encrypted with KMS.

EBS Snapshots

  • Point-in-time backups stored in AWS-managed S3.
  • Incremental - only changed blocks are stored.
  • Can create volumes in any AZ or Region.
  • EBS Snapshot Archive reduces cost by 75%.
  • Recycle Bin protects against accidental deletion.
  • Fast Snapshot Restore (FSR) removes first-read latency.

Instance Store (Ephemeral Storage)

Some instance types have local NVMe SSD storage physically attached to the host.

  • Extremely high IOPS
  • Ephemeral - data is lost on stop, termination, or host failure
  • Data survives a reboot
  • Cannot be detached or reattached
  • Size is fixed by instance type

EC2 Hibernate

When you hibernate an instance, RAM is saved to the root EBS volume, and the instance stops. On restart, the RAM state is restored.

Requirements:

  • Root volume must be EBS and encrypted
  • RAM must be less than 150 GB
  • Available for On-Demand, Reserved, and Spot instances
  • Hibernation duration up to 60 days

EC2 security

Security groups

Act as stateful firewalls at the instance (ENI) level. Allow rules only.

Key pairs

Used for SSH on Linux and RDP access workflow on Windows.

IAM Roles for EC2

Never store access keys on EC2. Attach an IAM role via an instance profile.

Part 2: ELB - Elastic Load Balancing

What is ELB?

Elastic Load Balancing automatically distributes incoming traffic across multiple targets across one or more Availability Zones.

Key benefits:

  • High availability
  • Health checks
  • SSL/TLS termination
  • Elasticity
  • Integration with Auto Scaling, ACM, CloudWatch, WAF, and Route 53

EC2 Auto Scaling automatically adjusts the number of instances based on demand, health checks, or schedules.

ELB cost considerations

  • ALB: Hourly + LCU
  • NLB: Hourly + NLCU
  • Cross-zone data transfer: Free for ALB; charged for NLB and GWLB if enabled

Hidden costs to watch

  • Stopped instances still incur EBS and Elastic IP costs
  • Unattached EBS volumes are billed
  • EBS Snapshots are billed
  • Elastic IPs are billed when unattached or attached to stopped instances
  • Cross-AZ data transfer is charged

Part 6: Exam traps and edge cases

  1. gp3 vs. gp2 - gp3 is cheaper and decouples IOPS/throughput from size.
  2. Instance Store data is lost on stop.
  3. Spread placement group max 7 per AZ.
  4. ALB does not have a static IP.
  5. NLB cross-zone is disabled by default.
  6. EBS volumes are AZ-locked.
  7. T instances in Unlimited mode can get expensive.
  8. Auto Scaling health check grace period matters.
  9. Multi-Attach is io1/io2 only, same AZ only.
  10. Spot instances - 2-minute warning.
  11. SNI required for multiple HTTPS domains.
  12. Launch Templates vs. Launch Configurations - use Launch Templates.

Final mental model

Every EC2/ELB exam question can be broken down with this sequence:

  1. What is the workload? (CPU-bound, memory-bound, I/O-bound, GPU)
  2. What is the demand pattern? (Steady, variable, bursty, fault-tolerant)
  3. What protocol? (HTTP/HTTPS, TCP/UDP, appliance inspection)
  4. What availability is needed?
  5. What storage is needed? (Persistent, ephemeral, shared)
  6. What is the cost priority?
  7. What security is needed? (SGs, IAM roles, IMDSv2, encryption)

If you can answer these seven questions for any scenario, you can answer any EC2/ELB question on the SAA-C03.

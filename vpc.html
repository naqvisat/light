AWS VPC: The Complete Guide for SAA-C03

Everything you need to know about Virtual Private Cloud — every definition, every component, every connectivity pattern, edge cases, hard limits, cost considerations, and exam traps — in one place.

What is a VPC?

A VPC (Virtual Private Cloud) is a logically isolated virtual network inside your AWS account where you launch AWS resources. Think of it as your own private data center within AWS - you control the IP address range, create subnets, configure route tables, and manage network gateways.

Key facts:

  • A VPC is Regional - it spans all Availability Zones in a single Region, but cannot span multiple Regions.
  • When you create an AWS account, AWS creates a default VPC in every Region with a CIDR block of 172.31.0.0/16, along with a default subnet in each AZ, an internet gateway, a default route table, a default NACL, and a default security group.
  • You can have multiple VPCs per Region (default: 5, adjustable).
  • Only one internet gateway can be attached to a VPC at a time.

CIDR - the language of IP addressing

What is CIDR?

CIDR (Classless Inter-Domain Routing) notation is how you define IP address ranges. It looks like this: 10.0.0.0/16.

The number after the / is the prefix length - it tells you how many bits of the address are fixed (the network portion). The remaining bits are available for hosts.

How to read CIDR

CIDR Network bits Host bits Total IPs Usable IPs in AWS subnet
/16161665,53665,531
/2020124,0964,091
/24248256251
/282841611

Simple rule: Each additional bit in the prefix halves the number of addresses. A /24 has 256 IPs. A /25 has 128. A /26 has 64.

VPC CIDR rules

  • Minimum: /28 (16 IP addresses)
  • Maximum: /16 (65,536 IP addresses)
  • VPC CIDR must come from the private IP ranges defined in RFC 1918:
    • 10.0.0.0 - 10.255.255.255 (10.0.0.0/8)
    • 172.16.0.0 - 172.31.255.255 (172.16.0.0/12)
    • 192.168.0.0 - 192.168.255.255 (192.168.0.0/16)
  • You can also use publicly routable CIDRs, but this is uncommon and causes issues with internet routing.
  • You can add up to 5 secondary CIDR blocks to a VPC (adjustable up to 50). The secondary CIDRs do not have to be contiguous with the primary.

AWS reserved IPs in every subnet

In every subnet, AWS reserves 5 IP addresses - the first four and the last:

IP Purpose
.0Network address
.1VPC router
.2DNS server
.3Reserved for future use
LastBroadcast address (AWS does not support broadcast, but reserves it)

Example: In a 10.0.1.0/24 subnet (256 IPs), usable IPs = 256 - 5 = 251.

Exam trap: A /28 subnet has 16 IPs, minus 5 reserved = only 11 usable IPs. This is the smallest subnet you can create.

CIDR planning for peering and hybrid

If you plan to use VPC peering, Transit Gateway, or on-premises connectivity, your CIDR blocks must not overlap. Overlapping CIDRs prevent peering connections and cause routing conflicts with VPN or Direct Connect.

Best practice: Plan your CIDR allocation across your entire organization before creating VPCs. Use non-overlapping ranges for every VPC and every on-premises network.

Subnets - public vs. private

A subnet is a range of IP addresses within your VPC, confined to a single Availability Zone. A subnet cannot span multiple AZs.

What makes a subnet "public"?

A subnet is public if:

  1. Its route table has a route to an Internet Gateway (typically 0.0.0.0/0 -> igw-xxx).
  2. Instances in it have a public IP or Elastic IP.

If either condition is missing, the subnet behaves as private even if you call it "public."

What makes a subnet "private"?

A subnet is private if its route table does not have a route to an Internet Gateway. Instances in a private subnet cannot receive inbound traffic from the internet and cannot initiate outbound internet connections unless you provide a NAT Gateway, NAT Instance, or VPC endpoint.

Typical architecture

VPC: 10.0.0.0/16
├── AZ-a
│   ├── Public Subnet:  10.0.1.0/24  → route to IGW
│   └── Private Subnet: 10.0.2.0/24  → route to NAT GW
├── AZ-b
│   ├── Public Subnet:  10.0.3.0/24  → route to IGW
│   └── Private Subnet: 10.0.4.0/24  → route to NAT GW
└── AZ-c
    ├── Public Subnet:  10.0.5.0/24  → route to IGW
    └── Private Subnet: 10.0.6.0/24  → route to NAT GW

Internet Gateway (IGW)

An Internet Gateway is a horizontally scaled, redundant, highly available VPC component that allows communication between your VPC and the internet.

Key properties:

  • One IGW per VPC. You cannot attach multiple IGWs to a VPC.
  • The IGW is attached at the VPC level, not at the subnet level. It is the route table that determines which subnets can use it.
  • It performs network address translation (NAT) for instances that have a public IPv4 address - mapping between their private IP and their public IP.
  • No bandwidth limits, no availability concerns - it is fully managed by AWS.
  • No cost for the IGW itself. You pay only for data transfer.
  • It supports both IPv4 and IPv6.

To give an instance internet access, you need all of:

  1. An IGW attached to the VPC
  2. A route in the subnet's route table: 0.0.0.0/0 -> igw-xxx
  3. A public IP or Elastic IP on the instance
  4. Security group and NACL rules allowing the traffic

Egress-Only Internet Gateway

An Egress-Only Internet Gateway is for IPv6 only. It allows outbound IPv6 traffic from your VPC to the internet but prevents inbound internet-initiated IPv6 connections.

Think of it as: A NAT Gateway equivalent for IPv6 (since NAT is not used with IPv6).

High availability for NAT

A NAT Gateway is redundant within its AZ but not across AZs. If the AZ goes down, instances in private subnets in other AZs that route through that NAT Gateway lose internet access.

Best practice: Create one NAT Gateway per AZ in each public subnet, and configure each AZ's private route table to point to the NAT Gateway in the same AZ.

Cost consideration: Each NAT Gateway costs money (hourly + data processing). For dev/test environments, you might use a single NAT Gateway to save cost, accepting reduced availability.

Exam trap: NAT Gateways are created in public subnets but used by private subnets. Never place a NAT Gateway in a private subnet.

Route tables

A route table contains a set of rules (routes) that determine where network traffic is directed.

Key properties:

  • Every VPC has a main route table created automatically.
  • You can create custom route tables and associate them with specific subnets.
  • A subnet can be associated with exactly one route table at a time.
  • A route table can be associated with multiple subnets.
  • Routes are evaluated using longest prefix match - the most specific route wins.

Route table anatomy

DestinationTargetPurpose
10.0.0.0/16localTraffic within the VPC (always present, cannot be deleted)
0.0.0.0/0igw-xxxInternet traffic via IGW (public subnet)
0.0.0.0/0nat-xxxInternet traffic via NAT Gateway (private subnet)
pl-xxxvpce-xxxTraffic to S3/DynamoDB via gateway endpoint
10.1.0.0/16pcx-xxxTraffic to peered VPC
10.2.0.0/16tgw-xxxTraffic via Transit Gateway
192.168.0.0/16vgw-xxxTraffic to on-premises via VPN/Direct Connect

The local route is always present and cannot be modified or removed. It ensures all traffic within the VPC CIDR stays within the VPC.

Route propagation

When you have a Virtual Private Gateway (VGW) attached for VPN or Direct Connect, you can enable route propagation on a route table. This automatically adds routes learned from your on-premises network via BGP.

Security: Security Groups vs. NACLs

Security Groups (stateful firewall)

A security group acts as a virtual firewall at the instance (ENI) level.

PropertyDetail
LevelInstance (ENI) level
StateStateful - if inbound traffic is allowed, the response is automatically allowed
Default behaviorAll inbound denied, all outbound allowed
RulesAllow rules only - you cannot create deny rules
EvaluationAll rules evaluated together (order does not matter)
Applies toOnly instances (ENIs) assigned to it
ScopeVPC-wide (but instances must be explicitly assigned)

Referencing other security groups: You can create rules that reference another security group instead of an IP range. This is the recommended approach for controlling traffic between tiers.

Network ACLs (stateless firewall)

A NACL acts as a firewall at the subnet level.

PropertyDetail
LevelSubnet level
StateStateless - inbound and outbound rules are evaluated independently; return traffic must be explicitly allowed
Default behaviorDefault NACL allows all inbound and outbound; custom NACLs deny all by default
RulesAllow AND deny rules
EvaluationRules evaluated in number order (lowest first); first match wins
Applies toAll traffic entering or leaving the subnet
ScopeVPC-wide (associated with subnets)

Ephemeral ports: Because NACLs are stateless, you must allow ephemeral ports (1024-65535) for return traffic.

How services connect - the complete picture

EC2 accessing S3

Option 1 - Public route (not recommended): EC2 in a public subnet accesses S3 over the internet via the IGW, or from a private subnet via a NAT Gateway.

Option 2 - Gateway VPC endpoint (recommended): Create a gateway endpoint for S3 in the VPC. EC2 accesses S3 over the AWS private network. Free.

Option 3 - Interface VPC endpoint: Needed if traffic comes from on-premises, cross-Region, or through Transit Gateway.

Lambda accessing DynamoDB

If Lambda is NOT in a VPC: Lambda runs in AWS's managed VPC and can call DynamoDB's public API directly.

If Lambda IS in a VPC: It loses internet access. To reach DynamoDB, use either:

  1. Gateway VPC endpoint for DynamoDB (recommended, free)
  2. NAT Gateway

Exam trigger: "Lambda function in a VPC times out when accessing DynamoDB" - add a gateway VPC endpoint for DynamoDB.

Lambda accessing S3

Same pattern as DynamoDB above. If Lambda is in a VPC, create a gateway VPC endpoint for S3 in the VPC.

Lambda accessing other AWS services (SQS, SNS, Secrets Manager, etc.)

If Lambda is in a VPC and needs to access services other than S3/DynamoDB, use either:

  1. Interface VPC endpoint for the specific service
  2. NAT Gateway for internet access

EC2 accessing RDS

Both are typically in the same VPC and communicate over the local VPC network. Security is controlled by security groups.

S3 event triggering Lambda

S3 sends event notifications to Lambda via AWS's internal infrastructure. This does not involve VPC networking at all.

VPC Peering

VPC Peering creates a direct network route between two VPCs using private IP addresses.

Key properties:

  • Works within the same account or across accounts.
  • Works within the same Region or across Regions.
  • Not transitive.
  • CIDR blocks must not overlap.
  • You must update route tables in both VPCs.
  • Security groups can reference the peered VPC's security groups (same Region only).
  • No single point of failure; uses AWS infrastructure.

VPC Peering limitations

  • Not transitive
  • Cannot peer if CIDRs overlap
  • Cannot use a peered VPC's internet gateway, NAT gateway, or VPN gateway
  • Cannot create more than one peering connection between the same two VPCs
  • Maximum of 125 active peering connections per VPC
  • Edge-to-edge routing is not supported

Transit Gateway (TGW)

A Transit Gateway is a Regional network hub that connects VPCs, VPN connections, Direct Connect gateways, and other Transit Gateways.

Key properties:

  • Acts as a cloud router
  • Supports transitive routing
  • Regional but supports inter-Region peering
  • Supports multiple route tables for segmentation
  • Supports ECMP
  • Bandwidth per VPC attachment: up to 50 Gbps
  • Maximum 5,000 attachments per TGW
  • Can attach VPCs from different accounts using AWS RAM

AWS PrivateLink

AWS PrivateLink enables you to privately expose a service to other VPCs or accounts without VPC peering, internet gateways, or NAT.

How it works

  • Service provider creates a service behind a Network Load Balancer (NLB).
  • Service consumer creates an interface VPC endpoint in their VPC.
  • Traffic flows over the AWS private network via ENIs - never touches the internet.

Key benefits

  • No need for VPC peering or Transit Gateway.
  • No CIDR overlap issues.
  • Scales to many consumers without routing complexity.
  • Provider allowlists specific accounts.

VPC Flow Logs

VPC Flow Logs capture information about the IP traffic going to and from network interfaces in your VPC.

Key properties:

  • Can be enabled at VPC level, subnet level, or ENI level.
  • Logs are published to CloudWatch Logs, S3, or Kinesis Data Firehose.
  • Capture accepted traffic, rejected traffic, or all traffic.
  • Flow logs are not real-time.
  • Cannot be modified after creation.

DNS in a VPC

Amazon-provided DNS

  • AWS provides a DNS server at the VPC base IP + 2. Also accessible at 169.254.169.253.
  • enableDnsSupport: When true, instances can use the Amazon-provided DNS.
  • enableDnsHostnames: When true, instances in the VPC receive public DNS hostnames if they have public IPs.
  • Both must be true for VPC endpoints with private DNS to work.

Route 53 Resolver

For hybrid DNS:

  • Inbound endpoint: On-premises DNS resolvers forward queries to Route 53 Resolver.
  • Outbound endpoint: Route 53 Resolver forwards queries to on-premises DNS.

VPC sharing (AWS RAM)

Using AWS Resource Access Manager (RAM), the owner of a VPC can share subnets with other AWS accounts within the same Organization.

Key properties:

  • The participant account can launch resources in the shared subnet.
  • The participant does not have access to the VPC owner's other resources.
  • Security groups, NACLs, and route tables remain under the owner's control.
  • This reduces the number of VPCs needed across an Organization.

Cross-Region connectivity

VPC Peering (inter-Region)

  • Works across Regions with encrypted traffic.
  • Uses AWS backbone.
  • Data transfer charges apply.

Transit Gateway peering (inter-Region)

  • Two Transit Gateways in different Regions can peer.
  • Enables transitive routing across Regions.
  • Static routes required.

Direct Connect Gateway

Connects a single Direct Connect connection to VPCs in multiple Regions.

Key things to remember for cross-Region

  • VPC endpoints are Region-specific.
  • Security groups cannot reference security groups in another Region.
  • NAT Gateways, Internet Gateways, and VPN Gateways are all Regional.
  • Data transfer between Regions always incurs charges.

Network security services

AWS Network Firewall

A managed firewall service deployed inside your VPC for deep packet inspection, intrusion prevention, and web filtering.

  • Deployed in a dedicated firewall subnet.
  • Integrates with VPC route tables to intercept traffic.
  • Supports stateful and stateless rules, domain filtering, and IPS/IDS.

AWS WAF (Web Application Firewall)

Protects web applications from common exploits. Deployed on ALB, API Gateway, CloudFront, or AppSync - not directly on VPC.

AWS Shield

DDoS protection. Shield Standard is free and automatic. Shield Advanced provides enhanced protections.

Elastic IP addresses

An Elastic IP is a static, public IPv4 address that you can allocate and associate with instances or NAT Gateways.

Key properties:

  • Regional scope.
  • Default limit: 5 per Region.
  • Free while associated with a running instance or NAT Gateway; charged when unattached or attached to a stopped instance.
  • You can remap an Elastic IP to another instance instantly.

Elastic Network Interface (ENI)

An ENI is a virtual network card attached to an instance in a VPC.

Key properties:

  • Each instance has a primary ENI (eth0) that cannot be detached.
  • You can create and attach additional ENIs.
  • Each ENI has one primary private IP, one or more secondary private IPs, one Elastic IP per private IP, one public IP, one or more security groups, and a MAC address.
  • ENIs are bound to a specific AZ.
  • Use case: Separate management and application traffic.

Availability design patterns

Multi-AZ architecture (standard)

  • Public subnets in each AZ with ALB/NLB
  • Private subnets in each AZ with application instances (Auto Scaling)
  • Private subnets in each AZ with database (Multi-AZ RDS)
  • One NAT Gateway per AZ
  • Single IGW (inherently redundant)

Multi-Region architecture

  • Separate VPCs in each Region
  • Connected via Transit Gateway peering or VPC peering
  • Route 53 for DNS-based routing
  • S3 cross-Region replication for data

Scalability design patterns

Horizontal scaling

  • Use Auto Scaling Groups across multiple AZs in private subnets.
  • Front with Application Load Balancer in public subnets.
  • ALB distributes traffic across AZs automatically.

Network scaling

  • NAT Gateway scales automatically up to 100 Gbps.
  • Interface endpoints scale automatically up to 10 Gbps per AZ (burst to 100 Gbps).
  • Transit Gateway scales to 50 Gbps per VPC attachment.
  • For VPN, use ECMP with Transit Gateway to scale beyond 1.25 Gbps per tunnel.

Common exam traps and edge cases

  1. NAT Gateway goes in the PUBLIC subnet.
  2. VPC Peering is NOT transitive.
  3. Gateway endpoints cannot be reached from outside the VPC.
  4. Lambda in a VPC loses internet access.
  5. Security Groups can only ALLOW.
  6. NACLs are stateless.
  7. One IGW per VPC.
  8. CIDR cannot overlap for peering.
  9. Egress-Only IGW is for IPv6 only.
  10. Direct Connect is NOT encrypted.
  11. Default NACL allows all, custom NACL denies all.
  12. Disable source/destination check for NAT Instances.

Final mental model

Every VPC networking question on the exam can be broken down with this sequence:

  1. Where is the source? (EC2, Lambda, on-premises, another VPC, another account, another Region)
  2. Where is the destination? (S3, DynamoDB, RDS, the internet, on-premises, another VPC)
  3. Does it need to cross a boundary? (Subnet -> internet, VPC -> VPC, Region -> Region, AWS -> on-premises)
  4. What is the traffic path? (Route table -> IGW, NAT GW, VPC endpoint, VPN, Direct Connect, Transit Gateway, peering)
  5. What controls the traffic? (Security group, NACL, endpoint policy, route table, VPN encryption)
  6. What are the cost implications? (Gateway endpoint = free; NAT Gateway, interface endpoint, TGW = cost)
  7. What are the availability requirements? (Multi-AZ NAT Gateways, redundant Direct Connect, multiple tunnels)

If you can answer these seven questions for any scenario, you can answer any VPC question on the SAA-C03.

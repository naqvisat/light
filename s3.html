AWS S3: The Complete Guide for SAA-C03

Everything you need to know about Amazon Simple Storage Service — every storage class, every security mechanism, every replication pattern, every lifecycle rule, every hard limit, and every exam trap — in one place.

What is S3?

Amazon S3 (Simple Storage Service) is object storage built for storing and retrieving any amount of data from anywhere. It is not a file system (that is EFS) and not block storage (that is EBS). Objects in S3 are accessed via HTTP/HTTPS APIs, not mounted as drives.

Foundational properties:

  • Durability: 99.999999999% (11 nines)
  • Scalability: Virtually unlimited storage. No capacity planning needed.
  • Availability: 99.99% for S3 Standard (SLA: 99.9%).
  • Consistency: Strong read-after-write consistency for all operations.
  • S3 is a Regional service.
  • S3 is a global namespace - bucket names must be globally unique across all AWS accounts worldwide.

Core concepts

Buckets

A bucket is a container for objects. Every object lives in exactly one bucket.

Bucket rules:

  • Bucket names must be globally unique across all AWS accounts.
  • Bucket names must be 3-63 characters, lowercase, no underscores, cannot look like an IP address.
  • You can have up to 100 general purpose buckets per account (adjustable to 1,000).
  • Buckets are created in a specific Region.
  • You cannot nest buckets.
  • Buckets can be empty, but they cannot be deleted if they contain objects.

Objects

An object is a file plus metadata. It consists of:

  • Key - the full path/name of the object
  • Value - the data itself
  • Metadata - system and user-defined metadata
  • Version ID - if versioning is enabled
  • Tags - up to 10 tags per object

Object size limits:

  • Single object maximum: 5 TB.
  • Single PUT upload maximum: 5 GB.
  • For objects larger than 5 GB, you must use multipart upload.
  • Multipart upload is recommended for objects over 100 MB.

Prefixes (not folders)

S3 has a flat structure - there are no real folders or directories. The slash in a key is just part of the key name. This matters because S3 performance scales by prefix.

Lifecycle policies

Lifecycle policies automate transitioning objects between storage classes or deleting them after a specified period.

Transition actions

Standard → Standard-IA → Intelligent-Tiering → Glacier Instant → Glacier Flexible → Deep Archive
Standard → One Zone-IA → Glacier Flexible → Deep Archive

Transition rules:

  • Cannot transition back up to Standard using lifecycle rules.
  • Objects must be in Standard for at least 30 days before transitioning to Standard-IA or One Zone-IA.
  • You cannot transition directly from Standard-IA to One Zone-IA.
  • Transitions incur a per-request cost.

Expiration actions

Day 0: Upload → S3 Standard
Day 30: Transition → S3 Standard-IA
Day 90: Transition → S3 Glacier Flexible Retrieval
Day 365: Transition → S3 Glacier Deep Archive
Day 2555 (7 years): Delete

Lifecycle rules scope:

  • Entire bucket or filtered subset by prefix or tag
  • Separate rules for current and previous versions
  • Can clean up incomplete multipart uploads

Versioning

Versioning keeps multiple variants of an object in the same bucket.

How it works

  • Not enabled: Objects can be overwritten and deleted permanently.
  • Enabled: Every PUT creates a new version with a unique Version ID.
  • Suspended: New versions get a version ID of null.
  • Versioning cannot be disabled once enabled - only suspended.

Deletes with versioning

  • A DELETE without specifying a version ID inserts a delete marker.
  • A DELETE with a specific version ID permanently removes that version.
  • To truly delete a versioned object, you must delete all versions and all delete markers.

Cost implications

Every version is a full copy and incurs storage charges.

MFA Delete

MFA Delete requires multi-factor authentication to permanently delete an object version or change the versioning state of a bucket.

Key rules:

  • Can only be enabled by the root user via the CLI or API.
  • Requires versioning to be enabled.
  • Does not affect normal operations like PUT, GET, LIST, or soft-delete.

S3 Object Lock

Object Lock enforces a WORM (Write Once Read Many) model.

Requires versioning and must be enabled at bucket creation time.

Retention modes

ModeWho can delete/overwrite?Can retention be shortened?
GovernanceUsers with special permissionsYes, with permission
ComplianceNobody - not even the root userNo

Legal hold

A legal hold prevents deletion regardless of retention settings and has no expiration.

S3 Glacier Vault Lock

For legacy Glacier vaults, Vault Lock allows you to set a WORM policy that cannot be changed once locked.

Encryption

S3 supports encryption at rest and in transit.

Encryption in transit

{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "Bool": { "aws:SecureTransport": "false" }
  }
}

Encryption at rest

SSE-S3AWS manages everythingAWSDefault. Simplest.

Default encryption: Since January 2023, all new objects are encrypted with SSE-S3 by default.

SSE-KMS considerations: KMS request quotas can become a bottleneck at high throughput. Bucket keys reduce KMS API calls and cost.

Enforcing encryption

{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}

Access control - the layered system

1. IAM policies

Attached to IAM users, groups, or roles. Control what S3 actions the identity can perform.

2. Bucket policies

JSON resource-based policies attached to the bucket. Can grant or deny access to specific principals, including cross-account principals and anonymous users.

3. Access Control Lists (ACLs)

Legacy mechanism. AWS recommends disabling ACLs with Bucket Owner Enforced.

4. S3 Block Public Access

Account-level and bucket-level settings that override any policy or ACL that would grant public access.

SettingWhat it blocks
BlockPublicAclsNew public ACLs from being applied
IgnorePublicAclsExisting public ACLs from taking effect
BlockPublicPolicyNew public bucket policies from being applied
RestrictPublicBucketsRestricts public access granted through public policies

5. S3 Access Points

Each access point has its own DNS name, policy, and optional VPC-only origin.

6. Pre-signed URLs

A pre-signed URL grants temporary access to an object without requiring AWS credentials.

7. Pre-signed cookies

Used with CloudFront to grant access to multiple files.

Replication

Cross-Region Replication (CRR)

Replicates objects to a bucket in a different Region.

Same-Region Replication (SRR)

Replicates objects to a bucket in the same Region.

Replication rules

  • Versioning must be enabled on both source and destination buckets.
  • Replication is asynchronous.
  • Only new objects are replicated after the rule is created.
  • Delete markers are not replicated by default.
  • Permanent deletes are never replicated.
  • An IAM role is required for S3 to perform the replication.
  • You can replicate to buckets in different accounts.
  • You can change the storage class during replication.
  • Chaining is not supported.

S3 Replication Time Control (S3 RTC)

Provides an SLA of 15 minutes for 99.99% of objects to be replicated.

S3 Event Notifications

S3 can send notifications when certain events occur on your bucket.

Supported destinations:

S3 Transfer Acceleration

Uses CloudFront edge locations to accelerate uploads and downloads to or from S3.

Uses endpoint: bucketname.s3-accelerate.amazonaws.com.

S3 Select and Glacier Select

Allows you to run simple SQL queries directly on objects in S3 without downloading the entire object.

Supported formats: CSV, JSON, Parquet.

S3 static website hosting

S3 can host static websites (HTML, CSS, JS, images).

How to set up:

  1. Enable static website hosting on the bucket.
  2. Set index and error documents.
  3. Make objects publicly accessible.
  4. Access via the S3 website endpoint.

Key properties:

  • HTTP only - no HTTPS on S3 website endpoints.
  • For HTTPS, put CloudFront in front of the bucket.
  • CORS must be configured if loading resources across origins.

Modern secure pattern

  1. S3 bucket remains private.
  2. CloudFront distribution uses Origin Access Control (OAC).
  3. Bucket policy allows only CloudFront to read objects.
  4. Users access the CloudFront URL over HTTPS.

S3 and VPC - private access

Gateway VPC Endpoint (free)

Creates a route in your route table to access S3 privately from within a VPC without internet access. No cost.

Limitation: Cannot be accessed from on-premises, peered VPCs, or Transit Gateway.

Interface VPC Endpoint (AWS PrivateLink)

Creates ENIs with private IPs in your subnet. Can be accessed from on-premises via Direct Connect or VPN.

Cost: Hourly per AZ + per-GB data processing.

S3 endpoint policies

Both gateway and interface endpoints support endpoint policies that restrict which S3 actions and buckets can be accessed through the endpoint.

S3 performance

Baseline performance

  • 3,500 PUT/COPY/POST/DELETE requests per second per prefix.
  • 5,500 GET/HEAD requests per second per prefix.
  • No limit on the number of prefixes per bucket.

Multipart upload

  • Required for objects over 5 GB.
  • Recommended for objects over 100 MB.
  • Uploads parts in parallel.
  • Failed parts can be retried independently.
  • Abort incomplete multipart uploads with lifecycle rules.

S3 Byte-Range Fetches

Download specific byte ranges of an object in parallel or partially.

S3 Batch Operations

Perform bulk operations on billions of objects with a single request.

Use cases:

  • Copy objects between buckets
  • Replace tag sets
  • Modify ACLs
  • Invoke Lambda for each object
  • Restore objects from Glacier
  • Replicate existing objects (S3 Batch Replication)

S3 Storage Lens

Provides organization-wide visibility into S3 usage and activity.

  • 28+ usage and activity metrics
  • Can aggregate across the entire AWS Organization
  • Free tier and advanced tier

S3 security best practices

  1. Block Public Access at the account level.
  2. Enable versioning for critical buckets.
  3. Enable MFA Delete for compliance-critical buckets.
  4. Use SSE-KMS for auditable encryption; SSE-S3 for default encryption.
  5. Enable S3 access logging or CloudTrail data events.
  6. Use VPC endpoints for private access.
  7. Use bucket policies with conditions.
  8. Enable Object Lock for WORM compliance.
  9. Use IAM roles for applications accessing S3.
  10. Use S3 Access Analyzer to find buckets shared externally.
  11. Use Macie to discover and protect sensitive data.

Common exam traps and edge cases

  1. Replication does not replicate existing objects.
  2. Replication does not replicate deletes by default.
  3. Minimum storage duration charges.
  4. SSE-KMS can throttle at high throughput.
  5. S3 website endpoints are HTTP only.
  6. Pre-signed URLs inherit the creator's permissions.
  7. Lifecycle transition minimum 30 days to IA.
  8. Access logging to the same bucket = infinite loop.
  9. CRR requires versioning on both buckets.
  10. Object Lock must be enabled at bucket creation.
  11. Glacier retrieval charges.
  12. S3 Strong Consistency.

Final mental model

Every S3 exam question can be broken down with this sequence:

  1. What is the access pattern? (Frequent, infrequent, archive, unknown → storage class)
  2. How long must data be retained? (Lifecycle policy, Object Lock, minimum storage duration)
  3. Who needs access? (IAM policies, bucket policies, pre-signed URLs, cross-account, public)
  4. How is it protected? (Encryption type, versioning, MFA Delete, Block Public Access)
  5. Does it need replication? (CRR for DR/compliance, SRR for same-Region copies)
  6. What is the network path? (Internet, VPC endpoint, Transfer Acceleration, CloudFront)
  7. What are the cost implications? (Storage class, retrieval fees, minimum duration, data transfer)

If you can answer these seven questions for any scenario, you can answer any S3 question on the SAA-C03.
